## Architecture

This is a Next.js 16 App Router application for AWS AIF-C01 exam prep, deployed to **Cloudflare Pages** (NOT Vercel). The `.vercel/` directory in the repo is a build artifact generated by Cloudflare's Next.js build tooling, not evidence of Vercel hosting. Deployment is automatic on `git push` to `main` (GitHub repo connected to Cloudflare Pages). The app is fully client-side — there are no API routes, server actions, or server functions yet.

**Routing:**
- `/` — Home page with mode selection (Mock Test vs Flashcards)
- `/practice?mode=mock&feedback=end` — 65-question timed mock exam
- `/practice?mode=flashcard&feedback=immediate` — 10-question flashcard session
- `/resources/*` and `/help/*` — Static informational pages

**Core data flow:**
1. `src/data/questions.json` is the single source of truth (~500+ questions)
2. `src/helpers/question-engine.ts` exports `getRandomQuestions`, `calculateScore`, and the `Question` type. It prioritizes unseen questions before recycling seen ones.
3. `src/context/SessionHistoryContext.tsx` persists seen question IDs to `localStorage` under the key `aws-prep-seen-questions`, so unseen questions are served first across sessions.
4. `src/components/session/PracticeSession.tsx` is the main stateful component — it owns the session state (questions, answers, bookmarks, timer) and orchestrates `QuestionCard`, `ResultsView`, `Sidebar`, and `SessionHeader`.

**Session modes:**
- `feedbackMode === "immediate"` (flashcard): answer revealed after each question, requires "CHECK ANSWER" button for multi-select
- `feedbackMode === "end"` (mock): answers revealed only after session finishes; sidebar nav and 90-minute countdown timer via `Sidebar`

**Question schema** (`Question` interface in `question-engine.ts`):
```ts
{ id, text, options, correctAnswer: string | string[], explanation, topic, multiSelect? }
```
Multi-select questions have `correctAnswer` as `string[]` and `multiSelect: true`.

**Styling:** Tailwind CSS v4 with `tailwind-merge` and `class-variance-authority`. Font is Plus Jakarta Sans via `next/font/google`. Color palette is light mode with indigo (`indigo-500`/`indigo-600`) as the primary accent.

**UI primitives:** `src/components/ui/button.tsx` and `card.tsx` are shadcn-style components using Radix UI slot and `cva`.

## Adding questions

Place raw questions in `new_questions.txt` (repo root) in the expected format, then run:

```bash
node scripts/import-questions.js
```

The script parses the format `Q: ... / Answer Choices / A. ... / Answer: X / Explanation ...`, deduplicates by normalized question text, and appends to `src/data/questions.json`. Other utility scripts: `scripts/add_single_question.js`, `scripts/remove_question.js`, and Python scripts in `src/data/` for processing/deduplication.

## Project Plan: Mock-Test Paywall

### Goal
Monetize mock-test mode while keeping a free flashcard taster. Mock test = paid; flashcards = free; no user accounts.

### Core constraint
Free and paid content MUST be different questions. The paid set must NEVER be bundled into client JS or reachable by a non-paying visitor. (Current problem being fixed: all 209 questions live in `src/data/questions.json`, bundled client-side, and free flashcard mode can reach the whole bank — so as built there is nothing to sell.)

### Question bank split
One bank today (209 questions). Split into two NON-OVERLAPPING pools: FREE pool of 30 questions (powers flashcards, client-side bundling fine) and PAID pool of remaining 179 (powers mock tests, server-side only, never bundled). Bank will grow over time.

### Access model
One-time payment = one mock-test set. Token burns on generation. Generated set is resumable for 90 minutes (matching the exam timer) so a crash/refresh returns the user to the SAME set; no new set without new payment. The 90-min exam timer runs continuously; on expiry the session ends. Accepted v1 tradeoff: a long disconnect can cost a paid attempt — low-stakes at $4.99, revisit only if refunds become frequent.

### Shuffle / selection
Mock test = server-side random subset + order shuffle from the paid pool per request, using Fisher–Yates (NOT `array.sort` with `Math.random`). Flashcards = client-side Fisher–Yates over the free pool. Note: existing `getRandomQuestions` in `question-engine.ts` uses localStorage "seen" tracking and assumes one shared bank — must be untangled when the bank is split.

### Platform & tools
Next.js App Router on Cloudflare Pages. Currently fully client-side, no server functions. To add: Cloudflare Pages Functions (endpoints) + Cloudflare KV (token storage).

### Pricing
$4.99 one-time.

### UI / CTAs
Flashcard CTA → "Try for Free" (safe anytime). Mock-test CTA → "$4.99" (tentative; consider "Unlock Mock Test – $4.99") — build label + Stripe Checkout wiring TOGETHER in the Stripe step, never before.

### Build order (one step at a time, verify after each)

### Working principles
Small single-purpose tasks; read-only recon before edits; verify site works after each change; only move things server-side when protecting paid content.
