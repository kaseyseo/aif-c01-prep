## Architecture

This is a Next.js 16 App Router application for AWS AIF-C01 exam prep, deployed to **Cloudflare Pages**. The app is fully client-side — there are no API routes, server actions, or server functions yet. The `.vercel/` directory in the repo is a build artifact generated by Cloudflare's Next.js build tooling, not an indication of Vercel hosting.

**Routing:**
- `/` — Home page with mode selection (Mock Test vs Flashcards)
- `/practice?mode=mock&feedback=end` — 65-question timed mock exam
- `/practice?mode=flashcard&feedback=immediate` — 10-question flashcard session
- `/resources/*` and `/help/*` — Static informational pages

**Core data flow:**
1. `src/data/questions.json` is the single source of truth (~500+ questions)
2. `src/helpers/question-engine.ts` exports `getRandomQuestions`, `calculateScore`, and the `Question` type. It prioritizes unseen questions before recycling seen ones.
3. `src/context/SessionHistoryContext.tsx` persists seen question IDs to `localStorage` under the key `aws-prep-seen-questions`, so unseen questions are served first across sessions.
4. `src/components/session/PracticeSession.tsx` is the main stateful component — it owns the session state (questions, answers, bookmarks, timer) and orchestrates `QuestionCard`, `ResultsView`, `Sidebar`, and `SessionHeader`.

**Session modes:**
- `feedbackMode === "immediate"` (flashcard): answer revealed after each question, requires "CHECK ANSWER" button for multi-select
- `feedbackMode === "end"` (mock): answers revealed only after session finishes; sidebar nav and 90-minute countdown timer via `Sidebar`

**Question schema** (`Question` interface in `question-engine.ts`):
```ts
{ id, text, options, correctAnswer: string | string[], explanation, topic, multiSelect? }
```
Multi-select questions have `correctAnswer` as `string[]` and `multiSelect: true`.

**Styling:** Tailwind CSS v4 with `tailwind-merge` and `class-variance-authority`. Font is Plus Jakarta Sans via `next/font/google`. Color palette is light mode with indigo (`indigo-500`/`indigo-600`) as the primary accent.

**UI primitives:** `src/components/ui/button.tsx` and `card.tsx` are shadcn-style components using Radix UI slot and `cva`.

## Adding questions

Place raw questions in `new_questions.txt` (repo root) in the expected format, then run:

```bash
node scripts/import-questions.js
```

The script parses the format `Q: ... / Answer Choices / A. ... / Answer: X / Explanation ...`, deduplicates by normalized question text, and appends to `src/data/questions.json`. Other utility scripts: `scripts/add_single_question.js`, `scripts/remove_question.js`, and Python scripts in `src/data/` for processing/deduplication.

## Project Plan: Mock-Test Paywall

### Goal
Monetize mock-test mode while keeping a free flashcard taster.
- Mock test = paid product.
- Flashcards = free marketing taster.
- No user accounts.

### Core constraint (the whole project depends on this)
Free and paid content MUST be different questions. The paid question set must
NEVER be bundled into the client JS or reachable by a non-paying visitor.
Current state (problem being fixed): all 209 questions live in a single
src/data/questions.json (274 KB), bundled client-side for every visitor, and
free flashcard mode can reach the entire bank over repeated sessions. As built,
there is nothing to sell. This plan fixes that.

### Question bank split
- One bank today: 209 questions in src/data/questions.json.
- Split into two NON-OVERLAPPING pools:
  - FREE pool: 30 questions. Powers flashcards. Bundling client-side is fine —
    it's the giveaway.
  - PAID pool: remaining 179 questions. Powers mock tests. Lives server-side
    ONLY (Cloudflare Pages Function / KV). Never bundled, never sent to an
    unpaid client.
- Bank will grow over time (more questions added to the paid pool primarily).

### Access model
- One-time payment = one mock-test question set.
- Token burns on test GENERATION.
- The generated set is resumable for 90 minutes (matching the exam timer), so a crash/refresh/back-button returns the user to the SAME set within that window. A NEW set cannot be generated without a new payment.
- The 90-minute exam timer runs continuously (same as the real exam); if it expires, the session ends whether or not all questions are answered.
- Accepted tradeoff for v1: a long disconnect can eat part/all of a paid attempt. At $4.99 this is low-stakes; revisit (add a grace buffer) only if refunds become frequent.

### Shuffle / selection
- Mock test: server-side selection of a random subset from the paid pool +
  order shuffle, per request. MUST use Fisher–Yates, NOT array.sort(() =>
  Math.random() - 0.5) (that is biased).
- Flashcards: client-side shuffle of the free pool, Fisher–Yates.
- Realistic goal without accounts: low overlap by probability, not guaranteed zero repeats.
- Note: existing getRandomQuestions in question-engine.ts uses localStorage "seen" tracking and assumes ONE shared bank. This must be untangled when the
  bank is split — flashcard "seen" tracking applies to the free pool only; mock-test selection moves server-side.

### Platform & tools
- Next.js (App Router) on Cloudflare Pages (NOT Vercel — the .vercel/ artifacts are byproducts of Cloudflare's Next.js build tooling).
- App is currently fully client-side; no server functions exist yet.
- Backend pieces to add: Cloudflare Pages Functions (endpoints) + Cloudflare KV (token storage).

### Pricing
- $4.99 one-time.

### UI / CTAs
- Flashcard card CTA: "Try for Free" 
- Mock-test card CTA: "Unlock for $4.99" 
- Build the label AND its click→Stripe Checkout wiring TOGETHER in the Stripe step. Do NOT add the paid CTA before Checkout exists, or it points at nothing.

### Build order (one step at a time; verify site works after each)
1. Split the bank: create the 30-question free pool and the 179-question paid
   pool, no overlap. Move the paid pool server-side (Cloudflare Pages Function),
   fetched at runtime with server-side Fisher–Yates selection/shuffle. Point
   flashcards at the free pool (stays client-side). Adjust getRandomQuestions /
   localStorage "seen" logic for the split.
2. Gate the mock-test Function with a KV token check (+ the 90-minute resumable-set logic). Include a temporary manual way to insert a test token into KV to verify gating before Stripe exists.
3. Stripe Checkout + webhook (Pages Function): on payment, generate a token, write to KV, hand it to the buyer.
4. Wire the mock-test CTA to Stripe Checkout; finalize CTA label.

### Working principles
- Small, single-purpose tasks. Read-only reconnaissance before edits.
- Verify the site still works after each change before moving on.
- Only move things server-side when it protects paid content; free/public
  content stays client-side.
